- **Название проекта**: Сервис кредитного скоринга: предсказание вероятности дефолта заёмщика

**Краткое описание**:  
Проект реализует end-to-end сервис для оценки кредитного риска на основе табличных данных о заёмщиках. Система принимает анкетные данные и кредитную историю, возвращает вероятность дефолта, категорию риска и решение о выдаче кредита. Решение упаковано в Docker-контейнер с API на FastAPI, включает базовую наблюдаемость и безопасную работу с конфигурацией.

## 2. Постановка задачи и контекст

### 2.1 Предметная область и задача
- **Задача**: Бинарная классификация — предсказание вероятности серьёзной просрочки (90+ дней) в течение 2 лет.
- **Пользователь**: Кредитный аналитик или автоматизированная система принятия решений в финтехе.
- **Сценарий**: При поступлении заявки сервис возвращает скоринг, категорию риска и рекомендацию (approve/reject), что позволяет ускорить обработку заявок и снизить риск выдачи проблемных кредитов.

### 2.2 Формулировка в терминах ML
- **Входные данные**: 10 числовых признаков (возраст, доход, кредитная нагрузка, история просрочек и др.).
- **Целевая переменная**: `SeriousDlqin2yrs` ∈ {0, 1} (1 = дефолт).
- **Ограничения**:
  - Интерпретируемость: возможность объяснить решение через важность признаков.
  - Производительность: время инференса < 200 мс для real-time API.
  - Устойчивость к выбросам и пропускам в данных.

### 2.3 Целевые метрики
| Метрика | Значение | Обоснование |
|---------|----------|-------------|
| **ROC-AUC** | 0.809 | Стандартная метрика для бинарной классификации с дисбалансом классов |
| **Gini coefficient** | 0.618 | Индустриальный стандарт в кредитном скоринге (2×AUC−1) |
| **Average Precision** | 0.262 | Учитывает точность на миноритарном классе (дефолты) |
| **Inference latency** | < 50 мс | Требование к real-time сервису |

## 3. Данные

### 3.1 Источник данных
- **Датасет**: [Give Me Some Credit (Kaggle)](https://www.kaggle.com/c/GiveMeSomeCredit)
- **Тип**: Открытый, обезличенный, учебный датасет для кредитного скоринга.
- **Объём**: 150 000 записей (после фильтрации 1 записи с невалидным возрастом).

### 3.2 Структура данных
| Признак | Тип | Описание |
|---------|-----|----------|
| `SeriousDlqin2yrs` | int (0/1) | Целевая переменная: наличие серьёзной просрочки |
| `RevolvingUtilizationOfUnsecuredLines` | float | Отношение баланса к лимиту по кредитным картам |
| `age` | int | Возраст заёмщика |
| `NumberOfTime30-59DaysPastDueNotWorse` | float | Количество просрочек 30–59 дней |
| `DebtRatio` | float | Отношение ежемесячных долгов к доходу |
| `MonthlyIncome` | float | Ежемесячный доход (с пропусками ~20%) |
| `NumberOfOpenCreditLinesAndLoans` | float | Количество открытых кредитных линий |
| `NumberOfTimes90DaysLate` | float | Количество просрочек 90+ дней |
| `NumberRealEstateLoansOrLines` | float | Количество ипотечных кредитов |
| `NumberOfTime60-89DaysPastDueNotWorse` | float | Количество просрочек 60–89 дней |
| `NumberOfDependents` | float | Количество иждивенцев (с пропусками ~3%) |

### 3.3 Предобработка и EDA
**Основные шаги** (реализованы в `src/data/` и `src/features/`):
1. **Очистка**: фильтрация записей с возрастом <18 или >120 (1 запись).
2. **Импутация**: медиана для `MonthlyIncome`, константа 0 для `NumberOfDependents`.
3. **Трансформация**: `log1p` для признаков с экстремальным перекосом (`RevolvingUtilization`, `DebtRatio`, `MonthlyIncome`).
4. **Масштабирование**: `RobustScaler` для устойчивости к выбросам.
5. **Агрегация**: создание признака `TotalPastDue` из трёх коррелирующих признаков просрочек (|r| > 0.97).

**Ключевые наблюдения из EDA** (`notebooks/01_eda.ipynb`):
- Класс-дисбаланс: 93.3% недефолтов, 6.7% дефолтов (соотношение ~14:1).
- Пропуски: `MonthlyIncome` (19.8%), `NumberOfDependents` (2.6%).
- Экстремальный перекос финансовых признаков (skewness > 90).
- Высокая корреляция между признаками просрочек (мультиколлинеарность).

## 4. Модели и подходы

### 4.1 Базовые модели (baseline)
| Модель | ROC-AUC | Gini | Время обучения | Комментарий |
|--------|---------|------|----------------|-------------|
| LogisticRegression | 0.794 | 0.588 | 0.19 с | Линейная модель, интерпретируемая, но ограничена в нелинейных зависимостях |
| RandomForest | 0.766 | 0.531 | 4.69 с | Ансамбль деревьев, склонен к переобучению без тщательной настройки |

### 4.2 Улучшенные модели и эксперименты
**Финальная модель**: LightGBM (градиентный бустинг)
- **Гиперпараметры**: `n_estimators=100`, `class_weight='balanced'`, `random_state=42`.
- **Преимущества**:
  - Автоматическая обработка пропусков и выбросов.
  - Высокая производительность обучения (0.43 с).
  - Возможность извлечения важности признаков.

**Эксперименты** (`notebooks/02_modeling.ipynb`):
- Сравнение трёх моделей на одном пайплайне предобработки.
- Валидация на стратифицированной тестовой выборке (20% данных).
- Анализ важности признаков для интерпретации решений.

### 4.3 Нейросетевые модели
Не применялись, так как для табличных данных градиентный бустинг показывает сопоставимое или лучшее качество при меньшей сложности и лучшей интерпретируемости.

## 5. Экспериментальный протокол и результаты

### 5.1 Протокол
- **Разделение данных**: стратифицированный train/test split (80/20) с сохранением пропорции классов.
- **Метрики**: расчёт на тестовой выборке, не участвовавшей в обучении или подборе гиперпараметров.
- **Воспроизводимость**: фиксированный `random_state=42` во всех этапах.

### 5.2 Сравнение моделей
| Модель | ROC-AUC | Gini | Avg Precision | Время обучения |
|--------|---------|------|---------------|----------------|
| LightGBM | **0.809** | **0.618** | **0.262** | 0.43 с |
| LogisticRegression | 0.794 | 0.588 | 0.217 | 0.19 с |
| RandomForest | 0.766 | 0.531 | 0.202 | 4.69 с |

### 5.3 Выбор финальной модели
**Выбрана**: LightGBM  
**Обоснование**:
1. **Качество**: ROC-AUC = 0.809 превышает индустриальный бенчмарк 0.74 для кредитного скоринга.
2. **Производительность**: обучение за 0.43 с, инференс < 50 мс.
3. **Устойчивость**: автоматическая обработка пропусков и выбросов.
4. **Интерпретируемость**: возможность извлечения feature importance для отчётности.

**Trade-offs**:  
LogisticRegression проще интерпретировать, но теряет ~1.5% качества. RandomForest требует больше ресурсов и склонен к переобучению.

## 6. Архитектура решения и сервис

### 6.1 Архитектура пайплайна
```
[Запрос API] 
    ↓
[Валидация Pydantic] 
    ↓
[Предобработка: импутация → log1p → RobustScaler] 
    ↓
[LightGBM: predict_proba] 
    ↓
[Бизнес-логика: категоризация риска] 
    ↓
[Ответ JSON]
```

### 6.2 API и endpoints
| Endpoint | Метод | Описание | Статус-коды |
|----------|-------|----------|-------------|
| `/health` | GET | Проверка работоспособности сервиса | 200, 503 |
| `/predict` | POST | Предсказание кредитного скоринга | 200, 400, 422, 500 |

**Пример запроса `/predict`**:
```json
{
  "RevolvingUtilizationOfUnsecuredLines": 0.15,
  "age": 35,
  "NumberOfTime30-59DaysPastDueNotWorse": 0,
  "DebtRatio": 0.3,
  "MonthlyIncome": 5000,
  "NumberOfOpenCreditLinesAndLoans": 3,
  "NumberOfTimes90DaysLate": 0,
  "NumberRealEstateLoansOrLines": 1,
  "NumberOfTime60-89DaysPastDueNotWorse": 0,
  "NumberOfDependents": 2
}
```

**Пример ответа**:
```json
{
  "default_probability": 0.3011,
  "risk_category": "medium",
  "decision": "approve_with_conditions",
  "model_version": "1.0.0",
  "request_id": "c2156234-9480-4916-b659-c1b39c9df33e"
}
```

## 7. Наблюдаемость, конфигурация и безопасность

### 7.1 Логи и наблюдаемость
- **Логирование**: структурированные логи с `request_id`, временем обработки, статусом запроса.
- **Health-check**: endpoint `/health` возвращает статус загрузки модели и версию сервиса.
- **Метрики**: консольные логи с таймингами, возможность расширения через Prometheus (опционально).

### 7.2 Конфигурации
- **Файлы**: `configs/.env.example`, `src/service/config.py` (pydantic-settings).
- **Параметры**: путь к модели, хост, порт, уровень логирования.
- **Гибкость**: перенастройка без изменения кода через переменные окружения.

### 7.3 Безопасность
- **Секреты**: не коммитятся в репозиторий, используются переменные окружения.
- **Данные**: только открытый обезличенный датасет, сырые файлы исключены через `.gitignore`.
- **Контейнер**: запуск от non-root пользователя, минимальный базовый образ.

## 8. Ограничения и дальнейшая работа

### Ограничения текущего решения
1. **Дисбаланс классов**: метрика Precision для миноритарного класса остаётся низкой (0.262).
2. **Статическая модель**: отсутствует механизм онлайн-обучения или детекции дрейфа данных.
3. **Упрощённая предобработка**: не все гипотезы из EDA реализованы (например, SMOTE, feature engineering).

### Планы развития
1. **Улучшение модели**:
   - Поиск гиперпараметров (Optuna, GridSearch).
   - Калибровка вероятностей для точной настройки порога принятия решений.
2. **Улучшение сервиса**:
   - Добавление endpoint `/metrics` в Prometheus-формате.
   - Реализация кэширования предсказаний для повторяющихся запросов.
3. **Улучшение данных**:
   - Интеграция внешних источников (кредитные бюро, поведенческие данные).
   - Синтез данных для балансировки классов (SMOTE, GAN).
