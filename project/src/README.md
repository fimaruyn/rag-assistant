# Исходный код проекта

В этой директории содержится основной production-ready код сервиса кредитного скоринга. Код организован модульно, следует стандартам PEP8, использует статическую типизацию и структурное логирование. Архитектура разделена по зонам ответственности для обеспечения тестируемости, переиспользования компонентов и изоляции бизнес-логики от инфраструктуры.

## Структура

- `src/data/` — модули загрузки и валидации сырых данных.
  - `loader.py` — чтение CSV, нормализация имён колонок, приведение типов, фильтрация аномальных записей (например, возраст вне диапазона [18, 120]).
  - `validator.py` — декларативная схема валидации через `pandera` (проверка типов, диапазонов, допустимых категорий, обязательных полей).
- `src/features/` — пайплайн предобработки и инженерии признаков.
  - `preprocessing.py` — набор `sklearn`-трансформеров: импутация пропусков (медиана/константа), логарифмическое преобразование skewed-признаков, агрегация коррелирующих счётчиков просрочек в единый признак, масштабирование через `RobustScaler`.
- `src/service/` — точка входа и логика REST API.
  - `main.py` — приложение FastAPI, управление жизненным циклом (`lifespan`), эндпоинты `/health` и `/predict`, middleware для логирования и замеров времени обработки запросов.
  - `config.py` — типизированная конфигурация через `pydantic-settings` (чтение из переменных окружения и `.env.example`).
  - `schemas.py` — Pydantic-модели для строгой валидации входящих JSON-запросов и структурирования ответов.
- `src/__init__.py` — инициализация пакета Python.

## Запуск сервиса

Сервис запускается как ASGI-приложение. Рекомендуемый способ (через `uv` для изоляции зависимостей и ускорения итераций):

```bash
cd project
uv run uvicorn src.service.main:app --host 0.0.0.0 --port 8000 --reload
```

После запуска доступны:
- REST API: `http://localhost:8000`
- Swagger UI: `http://localhost:8000/docs` (интерактивная документация с возможностью тестирования эндпоинтов)
- ReDoc: `http://localhost:8000/redoc`

## Ключевые особенности реализации

- Управление моделью: Пайплайн предобработки и сериализованная модель (`LightGBM`) загружаются единожды при старте сервера через контекстный менеджер `lifespan`. Это исключает повторную десериализацию `joblib`-артефакта на каждый HTTP-запрос и снижает latency.
- Валидация на границах системы: Входящие запросы проверяются Pydantic-схемами. Несоответствие типов, пропуск обязательных полей или выход значений за допустимые диапазоны возвращает статус `422 Unprocessable Entity` с детализацией ошибки.
- Наблюдаемость: Все запросы логируются с уникальным `request_id`, временем обработки и статусом ответа. Endpoint `/health` позволяет оркестраторам (Docker, Kubernetes, CI) проверять готовность сервиса и статус загрузки модели.
- Конфигурация: Параметры (хост, порт, путь к артефакту модели, уровень логирования) вынесены в переменные окружения. Реальные значения хранятся в `.env`, шаблон с комментариями — в `configs/.env.example`. Секреты не попадают в репозиторий.

## Тестирование модулей

Код сопровождается юнит- и интеграционными тестами, расположенными в директории `tests/`. Тесты проверяют корректность трансформаций, валидацию схем и контракты API.

Запуск полного набора тестов:

```bash
uv run pytest tests/ -v --tb=short
```
