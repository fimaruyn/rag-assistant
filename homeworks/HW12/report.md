# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Датасет `S12-hw-dataset.csv` — почасовой временной ряд (4320 наблюдений, январь–июнь 2025). Прогнозируется переменная `target` на один шаг вперёд.
- Данные разбиты хронологически: train (70 %) / validation (15 %) / test (15 %).
- Сравнивались 4 подхода: B1 (naive-last), B2 (moving-average), B3 (Ridge на лаговых признаках), R1 (GRU на оконном представлении).
- Лучшим по val_MAE оказался **B3** (val_MAE = 4.8829); на test подтверждено: MAE = 4.4657.

## 2. Среда и воспроизводимость

- Python: 3.10+
- pandas / numpy / scikit-learn: стандартные версии Colab
- torch: стандартная версия Colab
- Устройство (CPU/GPU): cuda
- Seed: 42
- Как запустить: открыть `HW12.ipynb` и выполнить Run All.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: 1 час
- Горизонт прогноза: 1 шаг (1 час)
- Размер датасета: 4320 наблюдений
- Диапазон дат: 2025-01-01 00:00 — 2025-06-29 23:00
- Дополнительные внешние признаки: нет (только `date` и `target`)
- Комментарий: ряд обладает выраженной суточной сезонностью (пик в дневные часы, спад ночью) и восходящим трендом от января к июню. Явных выбросов и пропусков нет. Нестационарность по уровню затрудняет работу простейших baseline.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- train: первые 70 % (3024 наблюдений, до 2025-05-06 23:00)
- validation: следующие 15 % (648 наблюдений)
- test: оставшиеся ~15 % (648 наблюдений, с 2025-06-03 00:00)
- Random split некорректен, поскольку временной ряд автокоррелирован: при случайном разбиении в train попадают будущие наблюдения, создавая утечку информации и завышая метрики.

### 4.2. Признаки для baseline-моделей

- Лаги: lag_1, lag_7, lag_14, lag_24, lag_168
- Rolling-признаки: rolling_mean_7, rolling_std_7, rolling_mean_24 (рассчитаны по shift(1) — без утечки)
- Календарные признаки: hour, dayofweek, day, month, is_weekend
- Пропуски после лагов/rolling: удалены (dropna); первые 168 строк train теряются
- Масштабирование: StandardScaler, обучен только на train

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (naive-last):** прогноз = значение на предыдущем шаге (lag_1). Не требует обучения.
- **B2 (moving-average):** прогноз = скользящее среднее за последние 24 часов (сдвинутое на 1 шаг).
- **B3 (ridge-lag-features):** Ridge-регрессия (alpha=1.0) на 13 признаках (лаги, rolling, календарные). Масштабирование StandardScaler по train.
- **R1 (gru-forecast):** GRU (hidden=64, 1 слой) на окне 48 часов отмасштабированного target. Предсказание следующего значения.

Дополнительно:
- Основная метрика выбора: val_MAE
- Дополнительные метрики: RMSE, MAPE
- window_size для GRU: 48
- Batch size: 64
- Максимальное число эпох: 60
- Optimizer: Adam
- Learning rate: 0.001
- Лучшая модель сохранялась по минимуму val_MAE (early stopping, patience=8)

## 7. Анализ

Наивный baseline B1 (naive-last) оказался неожиданно приемлемым для почасовых данных, поскольку соседние значения ряда сильно коррелированы — ряд меняется плавно от часа к часу. Тем не менее, MAE = 6.44 оставляет пространство для улучшения.

Moving-average B2 показал худший результат среди всех подходов. Это объясняется тем, что скользящее среднее за 24 часа сглаживает суточный цикл и отстаёт от текущего значения, особенно при наличии тренда.

Ridge на лаговых признаках (B3) оказался лучшим подходом. Сочетание lag_1 (инерция), lag_24 и lag_168 (суточная и недельная сезонность), rolling-статистик и календарных признаков дало модели достаточно информации для точного прогноза.

GRU (R1) показала результат, близкий к Ridge, но не превзошла его. Вероятная причина — для одношагового прогноза почасового ряда с регулярной сезонностью линейная модель на хороших признаках уже покрывает основную структуру данных, а нелинейность GRU не даёт дополнительного прироста.

Тренд и сезонность чётко видны в данных: суточный цикл повторяется стабильно, а средний уровень растёт. Это отражается на прогнозе — модели, учитывающие lag_24 и lag_168, лучше отслеживают оба компонента.

Утечки данных могли бы возникнуть при: (1) обучении scaler на всём датасете, (2) случайном split, (3) rolling без shift, (4) использовании будущих экзогенных переменных.

Выбор модели по validation, а не по test, принципиален: test должен имитировать реальный деплой, где мы не знаем будущего. Многократная оценка на test привела бы к неявной подгонке.

Наиболее показательные ошибки моделей — промахи в пиковых дневных значениях и в точках перелома тренда.

## 8. Итоговый вывод

Для почасового ряда с регулярной суточной сезонностью и плавным трендом Ridge-регрессия на лаговых, rolling- и календарных признаках — оптимальный базовый подход: она проста, быстра и даёт сильные метрики.

Простой baseline на признаках достаточен, когда сезонность регулярна, тренд несложный и горизонт прогноза короткий (1 шаг).

Переходить к GRU и другим нейросетям имеет смысл при многошаговом прогнозе, наличии нескольких связанных рядов, сложных нелинейных паттернов или когда ручное конструирование признаков затруднено.

Главный вывод из работы: корректный temporal split и контроль утечек данных — обязательное условие валидной оценки моделей на временных рядах; без этого даже хорошие метрики не гарантируют работоспособность в продакшене.
