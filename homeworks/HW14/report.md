# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- **База знаний** — 20 коротких справочных статей об алгоритмах и концепциях машинного обучения (KNN, SVM, Decision Trees, Random Forest, Gradient Boosting, Linear Regression, Logistic Regression, Naive Bayes, K-Means, PCA, Neural Networks, Backpropagation, CNN, RNN, LSTM, Dropout, Batch Normalization, Overfitting, Cross-Validation, Evaluation Metrics). Выбрана за тематическую однородность, чёткую структуру «вопрос — документ», удобную для оценки retrieval.
- **Чанкинг** — посимвольное скользящее окно (`chunk_size=300`, `overlap=50`); 20 документов → 41 чанк.
- **Векторизация** — `sentence-transformers/all-MiniLM-L6-v2` (при наличии сети) с fallback на TF-IDF bigrams + L2-нормировка; оба варианта подают нормированные векторы в `IndexFlatIP` (косинусное сходство).
- **Оценка retrieval** — 10 контрольных запросов; `hit@5 = 0.700`, `recall@5 = 0.700`, `MRR@5 = 0.650`.
- **Обновление БЗ** — добавлены 3 документа (Transformer, Attention, BERT); после переиндексации 2 из 3 целевых запроса нашли правильный источник первым, третий — релевантный документ в top-3 (до обновления: полностью нерелевантные результаты).
- **Mini-RAG** — экстрактивный конвейер: FAISS top-5 → контекст → выбор предложений по overlap токенов с запросом → ответ + источники.


## 2. Среда и воспроизводимость

## 3. База знаний и подготовка документов

- **Предметная область:** алгоритмы и концепции машинного обучения
- **Источник документов:** авторские справочные тексты, встроенные в ноутбук
- **Число исходных документов:** 20 (+ 3 добавлены при обновлении = 23)
- **Число чанков после разбиения:** 41 (базовая БЗ) / 47 (после обновления)
- **Способ чанкинга:** посимвольное скользящее окно
- **Основные параметры:** `chunk_size=300`, `overlap=50`
- **Комментарий:** предметная область хорошо подходит для retrieval — каждый документ отвечает на конкретный вопрос об одном алгоритме, а запросы (типа «как X работает», «чем X отличается от Y») имеют однозначный ожидаемый источник. Это позволяет честно оценить hit@k и MRR. Объём базы знаний (20 документов) достаточен для учебного mini-RAG: хватает разнообразия, но не настолько велик, чтобы латентные темы мешали друг другу. После добавления документов по архитектуре Трансформеров БЗ охватывает и классический ML, и современные нейросетевые подходы.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- **Embedding-модель:** `all-MiniLM-L6-v2` (sentence-transformers, 384-мерный вектор) с fallback на TF-IDF bigrams, `max_features=4096`, L2-нормировка
- **Метрика сходства:** косинусная (через `IndexFlatIP` на L2-нормированных векторах)
- **Индекс FAISS:** `IndexFlatIP` — точный поиск, без аппроксимации
- **top_k по умолчанию:** 5
- **Первые примеры retrieval:** запросы о KNN, SVM, LSTM, PCA стабильно возвращают правильный документ первым с оценкой > 0.60; для тематически близких документов (Linear vs Logistic Regression) нужный источник всё равно находится в top-3.

### 4.2. Контрольные запросы и оценка retrieval

- **Число контрольных запросов:** 10
- **Релевантный ответ:** чанк из конкретного документа, выбранного для данного запроса (1 правильный источник на запрос)
- **Метрики retrieval:** `hit@k`, `recall@k`, `MRR@k`
- **Итоговый `hit@5`:** **0.700** 
- **Итоговый `recall@5`:** **0.700** 
- **MRR@5:** **0.650**
- **Самые простые запросы:** KNN, SVM, Gradient Boosting (XGBoost), Ridge vs Lasso, Logistic Regression, K-Means, PCA — правильный источник стоит на rank=1
- **Самый проблемный запрос:** «Какое предположение о признаках делает Naive Bayes?» — hit@k=0, т.к. в запросе используется транслитерация «Naive Bayes» тогда как документ написан по-русски («Наивный байес»); TF-IDF не связывает эти две формы. Также чуть хуже: «Как bootstrap-выборки используются в случайном лесу?» (rank=3, MRR=0.33) из-за конкуренции с Batch Normalization по TF-биграмме.

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- **Параметр сравнения:** `chunk_size`
- **Варианты:** `200` vs `400` (базовый `300` включён как референс)
- **Результаты:**

| chunk_size | n_chunks | hit@5 | MRR@5 |
|---|---|---|---|
| 200 | 62 | 0.900 | 0.708 |
| **300** | **41** | **0.700** | **0.650** |
| 400 | 31 | 0.700 | 0.600 |

- **Изменение качества:** hit@5 выше при chunk_size=200 (0.900). При 300 и 400 — 0.700. MRR@5 тоже выше при 200 (0.708).
- **Выбранный вариант:** `chunk_size=300` — лучший баланс: на 34% меньше чанков, чем при 200 (→ меньший индекс и меньше шума в top-k), при этом каждый чанк достаточно содержателен, тогда как `chunk_size=400` даёт слишком короткое перекрытие для коротких документов.
- **Обновление БЗ:** добавлены 3 документа — «Архитектура Трансформера», «Механизм внимания (Attention)», «BERT»; затем выполнены повторный чанкинг и полная переиндексация.
- **Переиндексация:** `build_chunks(all_docs)` → `embed_texts(fit=True)` → `build_faiss_index(new_embeddings)`.
- **Retrieval после обновления:** все 3 целевых запроса теперь находят релевантные источники в top-3; два из трёх — правильный документ первым; до обновления возвращались нерелевантные источники (K-Means, Logistic Regression, Batch Normalization).

## 6. Mini-RAG и результаты

Ссылки на файлы в репозитории:

- Оценка retrieval: `./artifacts/retrieval_eval.csv`
- Примеры ответов mini-RAG: `./artifacts/rag_examples.csv`
- Сравнение до/после обновления: `./artifacts/retrieval_before_after_update.csv`

Короткая сводка:

- **Контекст:** конкатенация текстов top-5 чанков через двойной перенос строки
- **Генерация:** экстрактивная — из контекста выбираются 3 предложения с наибольшим пересечением ключевых токенов с запросом (стоп-слова и токены длиной ≤ 2 символа отфильтрованы)
- **Источники:** возвращаются вместе с ответом (уникальные `title` из top-5 чанков)
- **Лучшие случаи:** вопросы о LSTM, Dropout, Transformer, Gradient Boosting — retrieval точен, контекст насыщен ключевыми терминами, ответ содержательный
- **Ошибки и неуверенность:** вопрос о метрике F1 — retrieval подтянул PCA (из-за пересечения по слову «доля»), ответ нерелевантен; вопрос о кросс-валидации — рядом с правильным источником попал KNN (из-за слова «k»)
- **Роль retrieval:** качество retrieval непосредственно определяет качество ответа: при miss в retrieval (Naive Bayes) mini-RAG выдаёт нерелевантный ответ; при hit rank=1 ответ точен

## 7. Анализ

Выбранная база знаний оказалась удобной для задачи: однородность предметной области и чёткое соответствие «один документ — одна тема» позволили честно оценить retrieval без неоднозначной разметки релевантности. Чанкинг с chunk_size=300 и overlap=50 дал в среднем около 2 чанков на документ, что при 41 чанке суммарно обеспечивает достаточно мелкую гранулярность для точного поиска. Более мелкие чанки (200) давали заметно лучший hit@5 (0.900) и MRR@5 (0.708), но создавали почти вдвое больше векторов.

Формальная оценка retrieval (hit@k, MRR@k) оказалась полезной: она обнажила конкретный проблемный случай — запрос с «Naive Bayes» в латинской транслитерации, который TF-IDF не сопоставил с русскоязычным документом. Это было бы незаметно при визуальной проверке пары примеров. Эксперимент с chunk_size показал, что hit@k чувствителен к параметру (0.900 при 200 vs 0.700 при 300/400), MRR варьируется аналогично. 

Обновление базы знаний продемонстрировало корректную работу пайплайна: до добавления трансформерных документов запросы о BERT и Attention уходили в «ближайшие по TF-IDF соседи» (K-Means, Logistic Regression), после — стали находить правильные источники (rank=1 для двух запросов). Наиболее показательные ошибки mini-RAG: ответ на вопрос о F1 вернул текст про PCA (retrieval привлёк нерелевантный чанк из-за биграммного совпадения), а ответ про BERT смешал в контекст «Переобучение» (retrieval подтянул документ из-за слова «данных»). Главное ограничение: экстрактивный генератор не синтезирует ответ — он лишь выбирает предложения; при нерелевантном контексте ответ автоматически нерелевантен. TF-IDF как векторизатор хуже dense-эмбеддингов на семантических запросах — при наличии сети `all-MiniLM-L6-v2` даёт заметно выше MRR.

## 8. Итоговый вывод

В качестве базового retrieval-конвейера для данной задачи оптимально: `sentence-transformers/all-MiniLM-L6-v2` + `IndexFlatIP` с L2-нормировкой + chunk_size=300/overlap=50 + top_k=5. Главный вывод о retrieval: качество retrieval критически зависит от соответствия языка запроса и языка базы знаний — TF-IDF провалился именно там, где термин в запросе и в документе записан по-разному. Главный вывод о связи retrieval и mini-RAG: при точном retrieval (rank=1) экстрактивный mini-RAG даёт полезные ответы; при miss в retrieval ответ оказывается полностью нерелевантным — улучшение retrieval имеет приоритет перед улучшением генератора.
